## Approved plan (summary)
Streaming fraud-detection pipeline on IBM TabFormer (24M txns):
CSV replay producer (contract-validated, PAN-tokenized, DLQ) → Kafka (Redpanda local / Azure Event Hubs cloud)
→ Spark Structured Streaming windowed features → Redis (online) + Delta Lake (offline)
→ XGBoost training (Databricks Asset Bundle-packaged, run locally) → Flask `/score` with measured p50/p95/p99
→ dbt (duckdb) fraud-ops marts → Terraform-provisioned Azure (Event Hubs Standard, ACR, Container Apps), tested teardown.

## Known issues
- **Host port 8000 can collide with other local projects (environment-local, not a repo bug):** during v1.6 verification, an unrelated project's container (`docker-app-1`, from `~/Development/Projects/Analytics/healthcare-sepsis-prediction`) plus an active SSH tunnel were bound to `:8000` on this machine. Worked around without touching the other project: a local, git-ignored/uncommitted compose override file remapping only `api`'s *host* port (container-internal port stays 8000, so nothing else in the stack needs to change) — `services.api.ports: !override` with the new host mapping (Compose ≥2.24 syntax; a plain second `ports:` list *appends* rather than replaces, which is the trap to avoid). Delete the override file after use; never commit it.
- **CDC-topic pre-create step matters if bypassing `demo.sh`:** `scripts/demo.sh` explicitly creates `bankdb.frauddemo.bank.card_transactions` before starting the CDC chain; a manual `docker compose up` of individual CDC services without that step leaves `cdc_transformer.py` logging transient `UNKNOWN_TOPIC_OR_PARTITION` for ~30-45s until Redpanda's single-broker metadata converges after topic creation. Pre-existing librdkafka/Redpanda behavior, not new in v1.6 — self-resolves once the topic exists; just don't skip `demo.sh`'s topic-creation step in normal use.
