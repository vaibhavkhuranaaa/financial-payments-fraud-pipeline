# ADR 0003: CDC ingestion — Debezium change feed replaces CSV replay as the headline ingest (v1.2)

- **Status:** accepted (2026-07-18, per ticket 11 / approved v1.2 roadmap)
- **Context:** v1.x ingested by replaying a CSV straight onto Kafka — fine for a demo, but real card issuers stream the *change feed of the system of record*: authorizations land in the OLTP database first, and downstream consumers (fraud scoring included) read the captured change log. v1.2 makes that the architecture: `src.bank.txn_writer` INSERTs transactions into `bank.card_transactions` (SQL is now the true system of record), SQL Server CDC captures the changes off the transaction log, a streamer emits them to Kafka in Debezium's envelope format, and a thin transformer re-emits contract-v1 events on the existing `transactions` topic so Spark, the scorer, and the API are untouched.

## Decisions

1. **Real SQL Server CDC on Azure SQL Edge, with a `sp_cdc_scan` pump replacing the missing Agent.**
   - Verified empirically before building: SQL Edge (ARM) *does* support `sys.sp_cdc_enable_db` / `sp_cdc_enable_table` and populates change tables correctly — but only when `sys.sp_cdc_scan` runs, because the image ships **no SQL Server Agent** (the service that runs the cdc capture job in real SQL Server; `sys.dm_server_services` reports it Stopped).
   - Decision: a tiny dedicated compose service (`cdc-scan`, `src/bank/cdc.py --scan`) loops `EXEC sys.sp_cdc_scan` every ~2s. This is the documented pattern for Agent-less SQL editions; in production Azure SQL / SQL Server the Agent does exactly this and the pump service simply wouldn't exist. The tradeoff (one extra demo-only container) buys the *real* CDC change-table mechanics — LSNs, capture instances, `cdc.*` tables — rather than a simulated feed.
   - Rejected: polling-based incremental source (rowversion/timestamp column) — the ticket's fallback. Not needed since real CDC capture works, and polling a data column would demo a weaker guarantee (no log-ordered capture, lost intermediate states).
2. **Streaming the change feed: a small Python CDC streamer emitting Debezium's envelope format — because Debezium itself cannot stream from SQL Edge.**
   - Found live, after the connector was deployed and RUNNING: Debezium's SQL Server streaming loop advances its position with `sys.fn_cdc_increment_lsn`, which is **CLR-backed**, and SQL Edge has CLR permanently disabled (`sp_configure 'clr enabled', 1` → error 15392 "not supported by this edition"). Result: the initial snapshot works, then every streaming iteration fails (`Common Language Runtime(CLR) is not enabled on this instance`) in an infinite retriable-error loop. Everything else in Edge's CDC surface works: `fn_cdc_get_min_lsn`, `fn_cdc_get_max_lsn`, `fn_cdc_get_all_changes_*`.
   - Decision: `src/pipeline/cdc_streamer.py` reads the same change tables in LSN-windowed batches via `fn_cdc_get_all_changes` and re-implements the one missing function — LSN increment is 10 bytes of big-endian arithmetic in Python. It emits **byte-compatible Debezium envelopes** (`before`/`after`/`op`/`source`, epoch-millis temporals, double decimals) onto the same topic, keyed by primary key like Debezium — verified by a round-trip unit test through the transformer. Its LSN offset is persisted in `bank.cdc_offsets` strictly after the producer flush (same commit-after-flush boundary as every other consumer; kill/restart verified live with zero duplicate `event_id`s downstream).
   - The real Debezium Connect service and its connector config stay in the repo under compose profile `debezium` (`docker/connect/bankdb-source.json`, registered idempotently by `src/pipeline/connect_register.py` via `PUT`): against full SQL Server / Azure SQL — where the Agent runs the capture job and CLR exists — it is a drop-in replacement for the streamer with zero downstream changes, which is exactly the story the identical topic contract is there to prove. Config stays CI-linted by unit tests either way.
   - Rejected: switching `bank-db` to full `mssql-server` under x86 emulation just to host Debezium — reverses ADR 0002's deliberate ARM-native choice and taxes every query in the demo to fix a demo-only limitation.
3. **Bank DB migrates from `master` to a dedicated `frauddemo` database.**
   - CDC is refused outright on system databases, so the v1.1 shortcut of living in `master` (chosen to avoid a CREATE DATABASE step) had to end. `src.bank.seed` now bootstraps `frauddemo` idempotently (AUTOCOMMIT connection to `master`, `IF DB_ID(...) IS NULL CREATE DATABASE`) before applying the schema; `BANK_DB_NAME` defaults change everywhere. This is also simply more production-honest — nobody runs a core-banking schema in `master`.
4. **Debezium Connect (`quay.io/debezium/connect`) + SQL Server source connector, config-as-code, registered idempotently (compose profile `debezium`).**
   - The connector config lives in `docker/connect/` and is registered by a one-shot service via `PUT /connectors/<name>/config` (PUT, not POST — re-running is an update/no-op). Secrets are substituted from the environment at registration time, never committed.
   - `decimal.handling.mode=double` and `time.precision.mode=connect` pin the JSON wire shapes (DECIMAL would otherwise arrive base64-encoded, DATETIME2 in scale-dependent epoch units) so the transformer's parsing is deterministic.
   - Note: Debezium 2.x images moved off Docker Hub — `debezium/connect:2.7` no longer resolves; `quay.io/debezium/connect:2.7` is the published multi-arch (arm64-capable) image.
5. **Envelope handling in one tested Python module, not SMTs.**
   - `src/pipeline/cdc_transformer.py` consumes the Debezium topic (`bankdb.frauddemo.bank.card_transactions`), unwraps the envelope (`op` c/r → map `after` to a contract-v1 event; u/d and tombstones are skipped — authorizations are immutable, updates/deletes on this table are a data error, logged), validates against the same JSON-Schema contract the replay producer uses, and produces to `transactions` keyed by `card_token` (preserving per-card ordering). Invalid events go to the existing DLQ.
   - Rejected: Kafka Connect SMT chains (ExtractNewRecordState + renames) — they could reshape the envelope but not derive contract fields, validate against the JSON-Schema contract, or route a DLQ; and their config-string programming is untestable next to a 50-line Python module with unit tests on every op-code path.
6. **Delivery semantics made explicit end-to-end: at-least-once with commit-after-flush, effectively-once in SQL.**
   - Every consumer commits its offset only *after* its output is durable: the CDC streamer persists its LSN to `bank.cdc_offsets` after the producer flush, the transformer commits Kafka offsets after its producer flush, and the scorer commits after its bank-DB flush (auto-commit disabled on both). A crash between flush and commit re-delivers events; the `event_id` PK dedupe absorbs them. There is deliberately **no** exactly-once claim anywhere upstream of that SQL boundary.
7. **Both ingest modes stay first-class:** `make demo` (replay profile) and `make demo-cdc` (cdc profile) share every downstream service. Replay remains the fastest cold-start demo and the Azure Event Hubs story; CDC is the production-real architecture. Contract validation happens at the system-of-record boundary in CDC mode (`txn_writer` validates before INSERT; rows that fail mapping are counted and logged rather than DLQ'd to Kafka — the DB, not the broker, is the authority on what entered the system).

## Cost
$0 — everything local (Debezium Connect is one more container, ~1GB image). No cloud resources; the Terraform demo module is unaffected.
